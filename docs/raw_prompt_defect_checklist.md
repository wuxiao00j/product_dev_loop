# Raw Prompt Defect Checklist

这份清单用于在正式派单前判断：

- 当前输入是不是“原始需求草稿”
- 它有哪些缺陷会阻止直接发给执行器
- 哪些问题必须先修正
- 哪些只是质量不足，但可以继续收口

这份清单默认用于：

- `Prompt Gate` 判定过程中
- 或 `Prompt Gate = REWRITE_REQUIRED` 之后的 rewrite 阶段

使用顺序建议：

1. 先用这份清单判断“能不能直接发”
2. 如果不能，先用 `templates/round_prompt_rewrite_template.txt` 收口
3. 收口后，再用 `templates/prompt_quality_checklist.md`
4. 最后再按 `docs/prompt_scoring_rubric.md` 打分

注意：

- 这份清单本身不等于 Gate 结果
- 当前输入仍需要拿到正式的 `Prompt Gate` 结论

## 1. 一票否决项

出现任一情况，都不要把草稿直接发给 Cursor、Claude Code 或 Codex。

### 1. 没锁定项目上下文

直接判否的情况：

- 没有项目名
- 多项目并行，但无法判断草稿属于哪个项目
- 只有文件路径，没有项目边界

为什么是一票否决：

- 会直接导致串项目、串文档、串验收口径

### 2. 没有轮次 / 阶段 / PRD 状态

直接判否的情况：

- 没写当前轮次
- 没写当前阶段
- 没写 `PRD.md` 是 `draft` 还是 `locked`

为什么是一票否决：

- 执行器无法判断这轮是在继续开发、补缺陷，还是应该先停下来等变更确认

### 3. 没判断任务类型，且无法稳定归类

直接判否的情况：

- 草稿同时混着 `build + fix + polish + unblock`
- 无法判断本轮主目标到底是什么

为什么是一票否决：

- 任务类型不清，后面的目标、边界、验收和测试都会跟着失控

### 4. 没把目标收成一个可控 round

直接判否的情况：

- 一轮塞多个模块、多个页面、多个接口、多个子目标
- 草稿更像 backlog 打包，而不是单轮任务
- 本轮最低闭环都说不清

为什么是一票否决：

- 一旦目标不收敛，就算字段补齐了，也只是“看起来完整”

### 5. 没写本轮不做

直接判否的情况：

- 没有 `不做`
- 只有“尽量别扩散”这种软提醒

为什么是一票否决：

- 没有反边界，就等于默认允许扩散

### 6. 没写验收标准

直接判否的情况：

- 没有验收项
- 只有“做完”“补齐”“完善一下”
- 验收无法从结果层判断

为什么是一票否决：

- 主 agent 无法判断是否完成，用户也无法稳定测试

### 7. 存在依赖不确定项，但没有降级规则或阻塞规则

直接判否的情况：

- 接口、权限、环境、数据源、第三方依赖明显不确定
- 草稿却没写依赖满足时怎么做、依赖不满足时怎么办

为什么是一票否决：

- 执行器会自己编临时方案，或者把阻塞误做成功能开发

### 8. 没写结构化回传

直接判否的情况：

- 只有“做完告诉我”
- 没要求已完成、未完成、关键文件、特殊状态、风险

为什么是一票否决：

- 回执无法进入 `PROGRESS.md`、`IMPLEMENTATION_NOTES.md` 和 `PROJECT_STATE.md`

### 9. 没写测试要求

直接判否的情况：

- 没有测试入口
- 没有操作步骤
- 没有测试数据、预期结果或失败判定

为什么是一票否决：

- 这轮结果不可验证

### 10. 没写 PRD 对照要求

直接判否的情况：

- 没要求说明对应哪些 PRD 条目
- 没要求区分已满足和未满足项
- 没要求判断是否达成本轮验收

为什么是一票否决：

- 代码做了什么，不等于符合 PRD

### 11. `PRD.md` 已 `locked`，但草稿仍把范围变化写成直接实现

直接判否的情况：

- 草稿里包含新增需求、验收变化或方向变化
- 却没有先进入 `CHANGE_PROPOSALS.md`

为什么是一票否决：

- 这会绕过 PRD 边界

## 2. 质量不足但仍可修正的问题

下面这些问题不一定需要退回用户，但必须先收口再发。

### 1. 项目上下文基本清楚，但落点还不够稳

表现：

- 有项目名
- 但没有说明当前状态或本轮目标来源

处理：

- 补上“上轮结论 / 当前状态 / 本轮目标来源”

### 2. 任务类型可以推断，但没显式写出来

表现：

- 明显是在修 bug 或补闭环
- 但草稿没写 `build / fix / polish / unblock`

处理：

- 主 agent 先判断主类型，再把类型写进 prompt

### 3. 目标已经接近单轮，但还混着顺手项

表现：

- 有主目标
- 但还夹着“顺便优化”“如果方便一起做”

处理：

- 主目标保留
- 顺手项挪到 `不做`、后续轮建议或变更提议

### 4. 有验收雏形，但还偏空泛

表现：

- 写了想要的结果
- 但还不能直接判断完成与否

处理：

- 改写成外部可观察、可判断的验收项

### 5. 有测试要求，但还不够可执行

表现：

- 只有“请给测试建议”
- 没有具体步骤和失败判定

处理：

- 补齐测试入口、步骤、数据、预期结果、失败判定

### 6. 有回传要求，但结构不完整

表现：

- 只要求说改了哪些文件
- 没要求未完成、风险、特殊状态、PRD 对照

处理：

- 补齐结构化回传模板

### 7. 有依赖说明，但没有明确降级边界

表现：

- 提到了接口或权限问题
- 但没写依赖不满足时是否允许降级、允许降到哪里

处理：

- 明确允许的临时方案、禁止扩展的范围和何时应直接阻塞

## 3. 快速判定方法

如果当前草稿满足以下任一情况，直接判定“不能直接发”：

- 命中任一一票否决项
- 看起来像功能清单或 issue 池
- 很长，但仍说不清“这轮只做什么”
- 带文件路径 / API / UI 细节，却没有本轮不做和验收

如果当前草稿没有一票否决项，但仍明显偏松：

- 先收口
- 不要因为“差不多能做”就直接发

## 4. 这份清单的目标

这份清单不是为了提高文档形式感，而是为了防止三件事：

- 把用户草稿直接发给执行器
- 把很长的功能清单误判成合格 prompt
- 把看起来详细但实际上不可控的派单送进开发环节
