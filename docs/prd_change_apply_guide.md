# PRD 变更回写指南

这份指南只处理一种情况：`CHANGE_PROPOSALS.md` 里的某条提议，已经被用户明确批准。

## 基本原则

- 未获批准前，不得修改 `PRD.md`
- 获批后，回写时必须保留可追溯性
- 不要只改结果，不记录“为什么改”

## 推荐顺序

### 1. 先在 `DECISIONS.md` 记录批准决定

至少写清：

- 批准日期
- 被批准的提议编号
- 批准内容
- 批准原因
- 影响范围

这样可以保留“为什么改”的来源。

### 2. 再更新 `PRD.md`

更新方式：

- 只回写已批准的内容
- 保留 `PRD 状态：locked`
- 在 PRD 中补一行可追溯说明，例如：
  - 关联提议：`CHANGE_PROPOSALS.md / 提议 02`
  - 关联决策：`DECISIONS.md / 决策 03`

不要把原来的变更原因完全覆盖掉。

### 3. 在 `PROJECT_STATE.md` 标记 PRD 已更新

至少写清：

- 哪条提议已获批
- PRD 已在何时更新
- 当前是否需要进入新一轮开发或测试

### 4. 必要时同步更新 `PROMPTS/` 和验收标准

如果这次变更影响：

- 当前开发任务范围
- 测试入口
- 验收标准

就需要同步更新：

- 新一轮 `PROMPTS/round-xx-*.txt`
- `TEST_GUIDANCE.md`
- `TEST_RESULTS.md` 的后续测试目标

## 推荐记录格式

最小链路建议保持为：

1. `CHANGE_PROPOSALS.md` 中存在原始提议
2. `DECISIONS.md` 中记录“已批准”
3. `PRD.md` 中写明关联提议和关联决策
4. `PROJECT_STATE.md` 中写明 PRD 已更新
5. 新一轮 `PROMPTS/` 与测试文档接上新范围

## 不要这样做

- 不要在用户未明确批准前直接改 PRD
- 不要只在 PRD 里改结果，不留下提议和决策来源
- 不要更新 PRD 后却忘了同步验收标准和后续 prompt
