# Prompt Generation Flow

这份文档把“怎么生成合格的收口型 prompt”写成可执行步骤。

目标不是把模板填满，而是先判断当前轮最适合做什么，再把这一轮写成可执行、可验证、可回传的 prompt。

## Step 1. 先读上下文，不要先写 prompt

生成本轮 prompt 前，先按这个顺序阅读：

1. `PROJECT_STATE.md`
2. `PRD.md`
3. 最近一轮 `PROGRESS.md`
4. 最近一轮 `IMPLEMENTATION_NOTES.md`

按需补读：

5. `ARCHITECTURE.md`
6. `CHANGE_PROPOSALS.md`
7. `IMPLICIT_REQUIREMENTS.md`
8. `DECISIONS.md`
9. `TEST_RESULTS.md`

如果这些文件里出现以下情况，先停止写 prompt，先补判断：

- 当前阶段和项目现状对不上
- PRD 状态不清楚
- 上轮结论不清楚
- PRD 与最近反馈明显冲突
- 本轮任务来源不明

## Step 2. 判断当前轮最适合做什么

不要默认“下一轮就是继续开发功能”。先判断当前最适合推进哪种工作：

- 如果 PRD 还没锁定，优先补清需求，不进入开发派单
- 如果当前轮存在明显未闭环项，优先补闭环，不要扩新范围
- 如果测试反馈指出实现缺陷，优先修复缺陷
- 如果只是发现新想法，而 PRD 已 `locked`，先记变更建议，不直接派开发
- 如果当前阻塞来自环境或依赖，prompt 应聚焦解除阻塞，而不是功能开发

判断标准只有一个：

- 这一轮最值得做的，是“最小代价下最能推进闭环的动作”

## Step 3. 先选任务类型

在提炼目标前，先判断这轮 prompt 的任务类型：

- `build`
- `fix`
- `polish`
- `unblock`

选择方式：

- 要补新闭环，用 `build`
- 要修明确问题，用 `fix`
- 要优化已有路径但不改范围，用 `polish`
- 要先解除阻塞，用 `unblock`

如果这一步选不出来，先不要继续写 prompt。

## Step 4. 从 PRD 和反馈里提炼本轮目标

本轮目标要从以下来源里选一个主来源：

- PRD 当前条目
- 测试反馈
- 上轮未完成项
- 已确认的实现缺陷

提炼方法：

1. 找出当前最阻碍闭环的一件事
2. 把它压缩成一个本轮能完成的小任务
3. 写成“这轮只解决什么”
4. 再补一句“做完后用户能直接看到什么变化”

提炼后要避免这些写法：

- 一轮塞多个大功能
- 同时包含开发、重构、优化、体验升级等混合目标
- 只写功能名，不写结果

按任务类型补一句：

- `build`：写最小闭环和关键数据流
- `fix`：写复现现象和修复目标
- `polish`：写可动范围与不可动范围
- `unblock`：写阻塞点和临时方案 / 正式方案

## Step 5. 划定本轮边界

边界至少分成三类：

- `必做`：这轮必须交付，否则不算完成
- `不做`：这一轮明确禁止扩散的内容
- `重点文件 / 模块`：这轮主要应该落在哪些地方

划边界时的判断方式：

- 与本轮目标不直接相关的，不做
- 不能帮助本轮验收通过的，不做
- 会引入新范围判断的，不做
- 会改变 PRD 定义的，不做

如果你写不出明确的 `不做`，通常说明本轮还没收口。

补充判断：

- `build` 的边界重点是“不扩功能”
- `fix` 的边界重点是“不借机重构”
- `polish` 的边界重点是“不改结构和定义”
- `unblock` 的边界重点是“不把解阻写成全面改造”

## Step 6. 定义验收标准

验收标准要写成“外部可判断”的句子，而不是实现动作。

写法建议：

- 界面类任务：写用户操作后可见的结果
- 接口类任务：写请求、返回和错误处理结果
- 修复类任务：写原问题如何不再出现
- 收尾类任务：写哪些假数据、stub、未接线状态必须消失

自检问题：

- 不看代码，只看结果，能不能判断是否完成？
- 这几条验收标准能不能覆盖本轮目标？
- 如果执行者只完成一半，会不会被清楚识别出来？

按任务类型补强：

- `build`：验收要确认最小闭环和关键数据流已走通
- `fix`：验收要确认原问题消失且没有明显回归
- `polish`：验收要确认优化可感知且没有借机改范围
- `unblock`：验收要确认阻塞已解除到可继续推进

## Step 7. 写结构化回传要求

结构化回传的目标是方便主 agent 直接收口，而不是让执行者自由发挥。

至少固定要求这些字段：

1. 已完成
2. 未完成
3. 关键代码信息
4. `stub / mock / 假数据 / 隐藏入口 / 未接线`
5. 风险与需确认点
6. 测试建议
7. PRD 对照

写法要求：

- 说明每一项都必须单独列出
- 明确不要只回复“已完成”
- 明确要写关键文件路径和关键逻辑职责

类型侧重点：

- `build`：强调关键链路和未接线状态
- `fix`：强调根因、修复点和回归风险
- `polish`：强调实际优化点和明确未动范围
- `unblock`：强调阻塞是否解除、临时方案还是正式方案

## Step 8. 写测试要求

不要把“测试建议”写成一句空话。应明确要求执行者输出：

- 测试入口
- 操作步骤
- 测试数据
- 预期结果
- 失败判定

如果本轮是修复类任务，还要补一句：

- 请说明如何验证原问题已消失，且没有引入新的明显回归

如果本轮是 `unblock`，还要补一句：

- 请说明阻塞现在是否已解除到可以进入下一步，以及还剩哪些外部条件

## Step 9. 加入 PRD 对照要求

PRD 对照不是可选项。

prompt 里要明确要求执行者回答：

- 这轮对应的是哪些 PRD 条目
- 哪些条目已满足
- 哪些条目仍未满足
- 当前是否达到本轮验收标准

如果 PRD 已 `locked`，还要明确：

- 不得擅自修改 PRD 定义
- 发现偏差只能提出变更建议

## Step 10. 加入变更建议与隐性要求处理规则

在 prompt 里明确写清这几个规则：

- 新需求：先记为变更建议，不直接实现
- PRD 缺口：先指出，不直接改 PRD
- 隐性要求：先标记并请求确认
- 范围变化：先进入 `CHANGE_PROPOSALS.md`

这样可以避免执行者在实现过程中偷偷改范围。

## Step 11. 结合反馈判断这轮到底该不该继续派单

写 prompt 前，再问一次：

- 这是实现缺口，还是新需求
- 这是 bug，还是隐性偏好
- 这是可继续处理的问题，还是应该阻塞等待

如果判断结果是：

- 新需求：先走 `CHANGE_PROPOSALS.md`
- 隐性偏好：先确认，再写 `IMPLICIT_REQUIREMENTS.md`
- 已达到本轮目标：先测试或收口，不继续扩功能

## Step 12. 输出前用 checklist 自检

写完 prompt 后，不要直接发。先用 `templates/prompt_quality_checklist.md` 自检。

至少确认：

- 是否写明项目名、轮次、阶段、PRD 状态
- 是否要求执行前先读上下文
- 是否写清本轮必做和本轮不做
- 是否写了验收标准
- 是否要求结构化回传关键代码信息
- 是否要求说明 `stub / mock / 假数据 / 未接线`
- 是否要求给测试建议
- 是否要求做 PRD 对照
- 是否禁止擅改 PRD
- 是否写了新需求和隐性要求的处理规则

只要 checklist 有明显空项，就不要把这个 prompt 当成可派单版本。

## Step 13. 再按评分规约打分

自检之后，再按 `docs/prompt_scoring_rubric.md` 打分。

建议标准：

- 低于 `13` 分：不要直接发
- `13` 到 `15` 分：可直接发
- `16` 分及以上：高质量收口型 prompt

## 最终产出标准

最终生成的 prompt 应该满足这三个标准：

1. 执行者看完后知道这轮只做什么，以及明确不做什么
2. 主 agent 拿到回传后能直接判断是否过关
3. 即使多人并行项目，也不会因为 prompt 过松而串上下文、串范围、串验收口径
