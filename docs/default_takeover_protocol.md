# Default Takeover Protocol

这份文档定义：当用户只表达“继续项目 X / 接手项目 X / 跟进项目 X”这类意图时，`product_dev_loop` 默认应该怎么接管。

目标不是做复杂意图识别，而是把一句常见用户话术，稳定转成一套最小接管动作。

## 一句话定义

只要用户是在表达“继续某个项目”，而没有先指定更细任务，agent 默认先进入接管模式，再决定下一步动作。

## 触发语义

当用户表达以下意图时，应默认进入接管模式：

- 继续项目 X
- 接手项目 X
- 用 `product_dev_loop` 跟进项目 X
- 看看项目 X 现在到哪了
- 继续这个项目
- 跟进这个项目下一步

这里的重点不是字面完全匹配，而是用户是否在表达：

- 先恢复项目上下文
- 再决定下一步推进方向

## 默认动作顺序

如果用户没有额外指定更细任务，默认执行：

1. 识别项目名 / 项目标识 / 项目目录
2. 如果项目或范围存在关键歧义，先用 `templates/takeover_clarification_template.md` 做最小补问
3. 进入接管模式
4. 先读取 `docs/agent_handoff_quickstart.md`
5. 再读取该项目的 `PROJECT_STATE.md`
6. 再读取 `PRD.md`
7. 再读取 `PROGRESS.md`
8. 再按当前状态补读：
   - `IMPLEMENTATION_NOTES.md`
   - `TEST_RESULTS.md`
   - `TEST_GUIDANCE.md`
   - `ROUND_INDEX.md`
9. 形成接管摘要
10. 再判断下一步更适合：
   - 继续派单
   - 进入测试
   - 等待用户确认
   - 阻塞等待
   - 进入 `CHANGE_PROPOSALS.md`

## 默认输出要求

接管后第一轮输出应优先给出：

- 当前项目阶段
- PRD 是否 `locked`
- 当前轮次 / 最近轮次
- 最近一次关键进展
- 最近一次测试 / 反馈结论
- 当前最关键未闭环点
- 建议下一步动作

这轮默认输出的目的不是开始做事，而是先把“现在到底在哪”说清楚。

## 默认禁止项

在接管判断完成前：

- 不擅自改 `PRD.md`
- 不扩范围
- 不直接开始写下一轮开发 prompt
- 不把执行器“已完成”当成已闭环事实
- 不跳过 `PROJECT_STATE / PRD / PROGRESS` 直接看 prompt

## 如果用户已经给了更细任务

如果用户已经明确指定更细任务，例如：

- “根据 `TEST_RESULTS` 生成下一轮 prompt”
- “根据当前回执更新 `IMPLEMENTATION_NOTES.md`”
- “给这个项目写测试说明”

也不要跳过接管。

正确动作是：

1. 先做最小接管
2. 再立刻进入用户指定任务

这里的“最小接管”通常至少包括：

- `PROJECT_STATE.md`
- `PRD.md`
- `PROGRESS.md`

必要时再补：

- `IMPLEMENTATION_NOTES.md`
- `TEST_RESULTS.md`

## 多项目歧义处理

如果项目名不唯一、项目目录不明确、或存在多个候选项目：

- 应先向用户补问
- 不得直接猜测
- 不得在歧义状态下默认推进
- 默认一次只问最影响继续推进的那个问题

## 异常场景默认处理

### 1. 找不到项目目录

默认处理动作：

- 先确认是不是项目名写错
- 再确认项目是否还没初始化
- 不直接假设一个目录继续推进

### 2. 项目名对应多个候选

默认处理动作：

- 列出候选项目
- 请用户确认具体项目
- 在确认前不进入项目判断

### 3. `PROJECT_STATE.md` 与其他文件明显冲突

默认处理动作：

- 先标记冲突
- 以 `PROJECT_STATE.md + PRD.md + 最近测试/记录` 交叉核对
- 如果冲突影响下一步判断，先在接管摘要中说明，不直接推进

### 4. PRD 状态不明确

默认处理动作：

- 先查看 `PRD.md` 与 `PROJECT_STATE.md`
- 如果仍不清楚，明确标成“状态待确认”
- 在状态明确前，不改范围，不写新需求 prompt

### 5. 最近没有 `TEST_RESULTS.md`

默认处理动作：

- 不代表项目没问题
- 先用 `IMPLEMENTATION_NOTES.md` 和 `PROGRESS.md` 建当前状态
- 在接管摘要里标出“最近缺少正式测试结果”

### 6. 记录明显滞后于当前实现

默认处理动作：

- 先标记“记录可能滞后”
- 不直接用旧记录下新结论
- 优先补看最新回执或最近轮 prompt

### 7. 必须向用户补问的情况

这些情况不要继续推断：

- 项目不唯一
- 当前用户到底想继续哪个项目不明确
- 当前项目是否要继续当前范围还是切到新需求不明确
- 关键文件冲突已经影响下一步判断

如果用户回复很短，也不要再次追问一串背景。默认把那条回复当作最小确认，然后继续接管。

## 默认接管协议的最小成功标准

只要用户说“继续项目 X”，agent 至少应该做到：

1. 先接上当前项目状态
2. 先给出结构化接管摘要
3. 再判断现在该继续改、先测试、先确认还是先停

如果一上来就直接写下一轮 prompt，说明协议没有真正生效。
