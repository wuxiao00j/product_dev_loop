# Test Guidance Generation Guide

这份文档定义：主控 agent 如何根据执行器回执，生成“用户能直接照着测”的测试说明。

重点：

- 测试说明的目标读者不是开发者
- 测试说明不是代码说明
- 测试说明要让产品、老板、测试者直接拿去执行

## 测试说明的目标

一份合格的 `TEST_GUIDANCE.md`，应该做到：

- 用户不用看代码也知道怎么测
- 用户知道本轮最值得先验什么
- 用户知道哪些点执行器虽然说做了，但还需要重点验证
- 用户知道当前有哪些已知限制，不会误判结果

## 必写字段

至少覆盖：

- 测试入口
- 测试前提
- 操作步骤
- 测试数据
- 预期结果
- 失败判定
- 已知限制 / 暂不覆盖范围

还建议补：

- 本轮重点验证项
- 需重点验证的声称完成项

## 生成步骤

## Step 1. 先看本轮目标和验收标准

测试说明优先服务本轮目标，不是服务整个项目。

先回答：

- 这轮新增或修复了什么
- 哪些验收项最值得优先验证

## Step 2. 从回执里提取“用户能看到的结果”

优先提取：

- 页面变化
- 操作后结果
- 错误提示
- 数据是否保留
- 入口是否可见

不要优先提取：

- 函数名
- 文件名
- 实现策略

## Step 3. 优先写主路径和本轮新增路径

测试说明默认先写：

- 主路径
- 本轮新增路径
- 本轮修复路径

不要一上来写所有边角情况。

## Step 4. 如果证据不足，要明确“待验证”

如果执行器只说：

- “已实现”
- “应该可以”
- “已经接好了”

但没有给足够证据，测试说明里要明确提示：

- 该项需重点验证
- 不要默认视为闭环

例如：

- “执行器声称刷新后数据会保留，本轮请重点验证刷新后是否仍存在”

## Step 5. 显式标出特殊状态和限制

如果存在以下情况，必须在测试说明里写出来：

- `stub`
- `mock`
- `假数据`
- `未接线`
- 依赖外部条件
- 隐藏入口

因为这些内容直接影响用户怎么测、能测到什么程度。

## Step 6. 用用户语言，而不是开发语言

优先写：

- 打开哪个页面
- 输入什么
- 点击什么
- 应该看到什么
- 出现什么算失败

不要写成：

- 调用哪个函数
- 查看哪个状态
- 检查哪个 hook

## 好的测试说明长什么样

好的测试说明应具备这些特征：

- 可以直接复制给用户
- 步骤顺序清楚
- 预期结果外部可见
- 失败判定直接可识别
- 已知限制说得清楚

## 不好的测试说明长什么样

常见坏写法：

- “请验证功能是否正常”
- “如无异常则通过”
- “检查接口调用是否正确”
- “测试列表逻辑”

这些写法对非开发读者几乎没有帮助。

## 字段怎么写

## 测试入口

写用户能直接打开或触发的入口：

- 页面路径
- 操作入口
- 命令入口
- 必要账号或环境

## 测试前提

写开始前必须满足的条件：

- 环境是否已启动
- 是否需要登录
- 是否需要特定测试数据
- 是否依赖外部服务

## 操作步骤

写成线性步骤。

每一步只做一件事，避免合并太多动作。

## 测试数据

只写本轮最小需要的数据。

不要把大量无关样例塞进去。

## 预期结果

写用户肉眼或直接操作能判断的结果。

## 失败判定

直接写失败信号：

- 页面打不开
- 操作后没有结果
- 刷新后结果丢失
- 提示错误或状态不一致

## 已知限制 / 暂不覆盖范围

明确告诉用户：

- 哪些问题本轮不测
- 哪些能力本轮还没接好
- 哪些结果受外部条件影响

## 重点验证项

这里专门写：

- 本轮最关键的 1 到 3 个验证点
- 哪些是执行器声称已完成，但主控仍建议重点验的内容

## 一句话标准

测试说明不是“开发做了什么”，而是“用户现在按什么步骤能确认这轮是不是真的完成了”。
