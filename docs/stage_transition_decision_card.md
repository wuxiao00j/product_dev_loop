# Stage Transition Decision Card

这份文档是给主控 agent 的快速判断卡片，用来决定当前该：

- 继续派单
- 进入用户测试
- 等待用户确认
- 阻塞等待外部条件
- 进入 change proposal
- 进入下一阶段

先判断，再推进。

## 1. 继续派单

### 触发条件

- 本轮目标尚未达到验收标准
- 仍存在明确实现缺口
- 仍有 `stub / mock / 假数据 / 未接线` 阻碍闭环
- 问题属于可继续执行的实现问题，而不是范围变化

### 主控 agent 下一步该输出什么

- 下一轮建议：`继续派单`
- 为什么继续派
- 当前最关键未闭环点
- 下一轮应聚焦的单一目标
- 当前不建议做什么

### 哪些情况不应继续派单

- 反馈本质是新增需求
- 反馈是隐性偏好但尚未确认
- 关键外部条件缺失，代码侧继续做也无法闭环

## 2. 进入用户测试

### 触发条件

- 当前轮目标看起来已达到
- 没有阻断本轮结论的未完成项
- 执行器声称已完成，但仍需要用户或产品验证体验结果
- 存在可直接操作的测试入口

### 主控 agent 下一步该输出什么

- 用户可读测试说明
- 本轮重点验证项
- 哪些点虽然声称已完成，但仍需重点验
- 测完后需要用户反馈什么

### 哪些情况虽然还能改，但应该先让用户测试

- 主路径已经可用，只剩体验判断
- 结果看起来已满足 PRD，但缺少外部验证
- 风险主要在“是否真符合预期”，而不是“功能明显没做完”

## 3. 等待用户确认

### 触发条件

- 当前反馈可能改变范围
- 当前实现与 PRD 不一致，需要确认是修回去还是改需求
- 用户提出的是新增需求而不是 bug
- 隐性偏好尚未确认是否作为稳定要求

### 主控 agent 下一步该输出什么

- 建议：`先确认`
- 需要确认的具体问题
- 为什么不能直接推进
- 当前不建议做什么

### 必须先确认的情况

- `PRD.md` 已 `locked` 后的范围变化
- 会改变验收标准的用户意见
- 会导致下一轮类型切换的关键决策

## 4. 阻塞等待外部条件

### 触发条件

- 缺接口、环境、权限、账号、配置或上游决策
- 即使继续改代码，也不能真正验证结果
- 当前没有明确可执行的解阻动作

### 主控 agent 下一步该输出什么

- 建议：`阻塞等待`
- 当前阻塞点
- 缺少的外部条件
- 解阻前不建议做什么

### 哪些情况不应继续派单

- 只是为了“先做点什么”而开始无效改造
- 用临时绕路方案会直接改变已锁定范围

## 5. 进入 change proposal

### 触发条件

- 用户希望新增能力
- 用户希望改变范围、目标或验收标准
- 当前实现偏差如果要保留，会改变 PRD

### 主控 agent 下一步该输出什么

- 变更建议摘要
- 变更原因
- 影响范围
- 需要用户确认的点

### 哪些情况必须先进 change proposal

- `locked` PRD 下的新需求
- 会改变已有验收口径的改动

## 6. 进入下一阶段

### 触发条件

- 当前轮目标已完成
- 当前阶段产物已经齐全
- 下一步不再是继续实现，而是测试、反馈判断或新一轮计划

### 主控 agent 下一步该输出什么

- 当前阶段结论
- 下一阶段名称
- 进入下一阶段前的关键提醒

## “本轮已完成，但项目未完成” 的判断

这是最容易混淆的一类。

判断标准：

- 当前轮目标已达到验收标准
- 但 PRD 还有其他后续条目未做

这时应该写：

- 本轮结论：`通过`
- 项目状态：`项目未完成，继续按下一轮计划推进`

不要把“项目还没做完”误判成“本轮没完成”。

## 最快判断顺序

收到回执或测试反馈后，按这个顺序问：

1. 本轮目标达标了吗
2. 如果没达标，是实现问题、范围问题，还是外部阻塞
3. 如果达标了，是不是该先让用户测
4. 如果当前意见会改范围，是不是该先确认或走变更提议

## 一句话原则

不是“还能不能继续改”，而是“现在最正确的下一步是什么”。
