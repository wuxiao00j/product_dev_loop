# Task Type Prompt Guide

这份文档用于把 `Build Delegation` 阶段的 prompt 进一步按任务类型收口。

目的不是给任务贴标签，而是让主 agent 在写 prompt 时先判断：

- 这轮到底是在 `build`、`fix`、`polish` 还是 `unblock`
- 不同类型的 prompt，重点字段和验收口径不能混写

如果任务类型没选清楚，prompt 很容易退化成“泛泛做一下”。

## 先判断任务类型

优先用下面的判断方式：

- `build`：本轮要补一个新的最小闭环能力
- `fix`：本轮要修一个已经出现、可复现、会影响验收的具体问题
- `polish`：本轮要在不改范围的前提下优化已有实现的可用性、清晰度或体验
- `unblock`：本轮优先解除阻塞，让项目继续推进，而不是追求功能完整

如果一个任务同时像 `build` 又像 `fix`，优先看本轮主目标：

- 主要是在补新能力，就按 `build`
- 主要是在消除已知问题，就按 `fix`

## 1. build

### 适用场景

- PRD 里有明确条目尚未落地
- 当前轮需要补一个新的用户可感知能力
- 目标是做出一个最小可验证闭环，而不是铺很多壳子

### 核心目标

- 在本轮做出一个最小闭环
- 把主路径走通
- 说明数据从哪里来、怎样流、结果怎样被看到

### 必写字段

- 当前项目信息
- 本轮目标
- 本轮边界
- 数据流或关键链路说明
- 验收标准
- 测试要求
- PRD 对照
- 结构化回传

### 最容易漏掉的段落

- `不做`
- 数据流说明
- 本轮只做一个小闭环的约束

### 验收重点

- 主路径是否真正闭环
- 数据是否从输入流到结果，而不是只做 UI 壳子
- 本轮是否没有偷偷扩成下一轮内容

### 执行器回传重点

- 关键入口和关键文件
- 数据流经过了哪些环节
- 是否仍有 `stub / mock / 假数据 / 未接线`
- 哪个 PRD 条目已满足，哪个还没满足

### 常见坏写法

- “把 xx 页面做出来，顺便把相关功能都补齐”
- “先把 UI 搭起来，后面再说”

## 2. fix

### 适用场景

- 已有实现出现明确问题
- 用户测试、执行器回执或日志已经给出异常现象
- 本轮目标是修问题，不是继续加功能

### 核心目标

- 明确复现现象
- 指定修复目标
- 控制回归风险
- 要求修后验证

### 必写字段

- 当前项目信息
- 问题现象
- 复现路径
- 修复目标
- 本轮边界
- 回归风险提示
- 修后验证要求
- 结构化回传

### 最容易漏掉的段落

- 复现步骤
- 修后如何验证
- 哪些相关路径不能被修坏

### 验收重点

- 原问题是否不再出现
- 修复有没有引入新问题
- 是否只修当前问题，没有借机改一大片

### 执行器回传重点

- 原问题根因
- 修复点落在哪些文件
- 修后如何验证
- 哪些相关路径做了回归检查

### 常见坏写法

- “这个有点问题，你修一下”
- “顺便把相关逻辑重构一下”

## 3. polish

### 适用场景

- 主功能已经能用，但清晰度、可用性、文案、状态展示或交互细节仍不达标
- 本轮不是改范围，而是优化已有路径

### 核心目标

- 在不扩范围的前提下提升已有结果
- 明确可动范围与不可动范围
- 防止执行器借优化名义重构或扩功能

### 必写字段

- 当前项目信息
- 要优化的对象
- 可动范围
- 不可动范围
- 验收标准
- 测试要求
- PRD 对照
- 结构化回传

### 最容易漏掉的段落

- 不可动范围
- 明确禁止重构
- 优化完成的判定口径

### 验收重点

- 用户能感知到具体改善
- 不改变原有功能定义
- 不引入额外技术改造或目录改造

### 执行器回传重点

- 实际优化了什么
- 哪些地方明确没有动
- 是否存在为了优化而引入的新临时方案

### 常见坏写法

- “把体验优化一下”
- “顺手把结构整理一下更合理”

## 4. unblock

### 适用场景

- 当前轮卡在环境、依赖、配置、接口接入、权限、数据条件或决策缺失
- 目标是先让项目继续推进，而不是一次做到正式完整方案

### 核心目标

- 先解除阻塞
- 区分临时方案与正式方案
- 不扩散成全面改造

### 必写字段

- 当前项目信息
- 阻塞现象
- 阻塞原因或怀疑原因
- 本轮要解除的阻塞点
- 临时方案 / 正式方案说明
- 本轮不做
- 验收标准
- 结构化回传

### 最容易漏掉的段落

- 阻塞原因
- 临时方案与正式方案的区分
- 解阻后下一步是什么

### 验收重点

- 阻塞是否被解除
- 项目是否能进入下一步
- 是否没有把解阻扩展成大规模改造

### 执行器回传重点

- 当前阻塞是否真正解除
- 用的是临时方案还是正式方案
- 还剩哪些外部依赖或用户决策
- 下一轮最适合做什么

### 常见坏写法

- “先把这个卡点搞定，能优化的都一起优化”
- “为了避免再卡住，顺手重做整套方案”

## 按类型收口时的共通硬规则

- `PRD.md` 已 `locked` 时，不得擅改
- 多项目必须隔离，不串上下文
- 每轮都要有测试建议
- 每轮都要做 PRD 对照
- 新需求先走 `CHANGE_PROPOSALS.md`
- 隐性偏好先确认，再写 `IMPLICIT_REQUIREMENTS.md`

## 快速判断法

如果你已经写完 prompt，却还是说不清：

- 这轮是补新闭环、修问题、做优化还是先解阻
- 本轮为什么是这个类型而不是另一个类型

那说明任务类型还没收口，prompt 也还不够稳。
