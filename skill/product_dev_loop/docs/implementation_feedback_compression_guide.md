# Implementation Feedback Compression Guide

这份文档定义：当 Cursor、Claude Code 或 Codex 返回一大段实现说明后，主控 agent 应该如何压缩成项目内关键记录。

目标不是“整理漂亮摘要”，而是把长回执拆成不同用途的沉淀：

- 给自己快速判断当前轮结论
- 给项目保留最少但足够的长期记录
- 给用户保留能直接测试和决策的信息

## 总原则

- 不照抄执行器原话
- 不把执行器自我评价直接当事实
- 只保留能帮助下一轮判断和测试的关键内容
- 同一条信息按用途写入不同文件，而不是全塞进一个地方

## 先做三步过滤

收到长回执后，先做这三步：

1. 提取事实
2. 提取风险与未完成项
3. 提取用户需要验证的点

这里要先区分：

- `实现事实`：执行器明确改了什么、哪些文件、哪些逻辑、哪些状态仍是 `stub / mock / 假数据 / 未接线`
- `执行器自我评价`：例如“已经完整支持”“应该没问题”“大体可用”
- `可验证事实`：能通过测试入口、页面结果、接口结果确认的内容
- `待确认推测`：例如“理论上已经满足 PRD”“应该不会再有问题”

长期记录只优先保留前两类中的关键信息，不保留自我评价腔。

## 各文件分别该写什么

## 1. `PROGRESS.md`

这里写的是“当前最关键进展”，不是实现详情。

适合写进 `PROGRESS.md` 的内容：

- 本轮主要完成了什么
- 当前最关键未完成点是什么
- 当前轮结论是什么
- 当前下一步方向是什么

不适合写进 `PROGRESS.md` 的内容：

- 大段文件列表
- 具体函数名和细碎代码点
- 执行器原始过程解释
- 多段推理和自我辩护

压缩要求：

- 1 到 2 句短摘要优先
- 轮次记录只写目标、结果、结论
- 全局状态只保留闭环判断最需要的信息

## 2. `IMPLEMENTATION_NOTES.md`

这里写的是“本轮实现事实”，用于后续派单、判断和回看。

适合写进 `IMPLEMENTATION_NOTES.md` 的内容：

- 已完成项
- 未完成项
- 关键文件
- 关键文件职责
- 关键逻辑
- 风险
- `stub / mock / 假数据 / 隐藏入口 / 未接线`
- PRD 对照结果
- 当前结论

不适合写进 `IMPLEMENTATION_NOTES.md` 的内容：

- 原始回执整段复制
- 不重要的小修小改列表
- 执行器的情绪化描述
- “应该可以”“理论上没问题”这类未经验证的话

写法要求：

- 先写事实，再写判断
- 把“改了什么”和“为什么重要”分开写
- 标出哪些内容仍需测试确认

## 3. `TEST_GUIDANCE.md`

这里写的是“用户现在怎么测”，不是开发说明。

适合写进 `TEST_GUIDANCE.md` 的内容：

- 本轮重点验证项
- 测试入口
- 测试前提
- 操作步骤
- 测试数据
- 预期结果
- 失败判定
- 已知限制和暂不覆盖范围
- 哪些点是执行器声称已完成，但仍需重点验证

不适合写进 `TEST_GUIDANCE.md` 的内容：

- 函数名解释
- 文件路径堆砌
- 开发实现过程
- 调试日志

## 4. `TEST_RESULTS.md`

这里写的是“实际测到了什么”，不是实现计划。

适合写进 `TEST_RESULTS.md` 的内容：

- 测试日期
- 测试执行人
- 测试入口
- 实际结果
- 问题列表
- 问题分类
- 是否通过
- 下一步建议

不适合写进 `TEST_RESULTS.md` 的内容：

- 开发者推测
- 下一轮完整 prompt
- 与本轮测试无关的历史背景

## 5. `PROJECT_STATE.md`

这里写的是“项目当前状态判断”，不是本轮实现详情。

适合写进 `PROJECT_STATE.md` 的内容：

- 当前阶段
- PRD 状态
- 当前目标
- 最近结论
- 当前阻塞
- 下一步建议

不适合写进 `PROJECT_STATE.md` 的内容：

- 长文件清单
- 逐条实现细节
- 全量测试步骤

## 哪些信息不该长期保留

下面这些内容通常只需要短暂参考，不该大段进入项目记录：

- 执行器原话全文
- 工具运行过程
- 来回尝试失败的中间方案
- 与当前结论无关的长篇解释
- “我感觉应该没问题”“已经完整支持”这类自我评价

如果其中某条信息真的重要，应改写成短句事实再写入，而不是复制原文。

## 如何提炼关键字段

## 已完成项

只保留：

- 真正影响本轮闭环的完成项
- 可映射到验收标准或 PRD 条目的完成项

不要保留：

- 微小杂项
- 对闭环无帮助的顺手修改

## 未完成项

只保留：

- 阻碍本轮闭环的未完成项
- 会影响测试判断的未完成项

写法：

- 未完成什么
- 为什么没完成
- 是否影响本轮结论

## 关键文件

只保留：

- 与本轮目标直接相关的文件
- 决定后续派单和测试的人必须知道的文件

每个文件最好只写一行：

- 文件路径
- 文件职责

## 关键逻辑

只保留：

- 主路径如何工作
- 问题修在哪
- 数据如何流

避免写成代码注释复读。

## 风险

优先保留：

- 仍需测试确认的点
- 仍依赖外部条件的点
- 会影响用户判断的点

## `未接线 / stub / mock / 假数据`

必须单独标记。

因为这些状态直接影响：

- 是否能进入测试
- 是否能判定为通过
- 是否要继续派单

## 用户真正需要知道的测试点

优先提炼：

- 主路径是否真的闭环
- 本轮新增或修复的关键路径
- 执行器声称完成但仍缺证据的部分
- 因 `stub / mock / 未接线 / 外部依赖` 需要特别注意的地方

## 如何区分实现事实与执行器自我评价

实现事实常见形式：

- “新增了 `src/pages/todos.tsx` 中的筛选状态切换”
- “列表初始化仍使用假数据”
- “删除动作还未接线”

执行器自我评价常见形式：

- “功能应该已经完整了”
- “理论上没有问题”
- “现在已经比较完善”

处理规则：

- 事实可写入长期记录
- 自我评价只能作为参考，不直接入档
- 如果自我评价对应重要判断，必须改写成待验证表述

例如：

- 不写：“已经完整支持持久化”
- 改写为：“执行器声称已接入持久化，需在测试中重点验证刷新后结果是否仍存在”

## 如何区分可验证事实与待确认推测

可验证事实：

- 页面是否出现结果
- 提交后是否新增
- 刷新后数据是否保留
- 某入口是否可直接访问

待确认推测：

- “大概已经符合 PRD”
- “应该不会回归”
- “理论上外部接口可用了”

处理规则：

- 可验证事实可直接进入测试说明与记录
- 待确认推测要明确标注 `待验证`
- 不要把推测写成结论

## 压缩时的推荐顺序

1. 先看回执里的 `已完成 / 未完成 / 特殊状态 / 测试建议 / PRD 对照`
2. 抽取 3 到 5 个真正影响判断的事实
3. 写 `IMPLEMENTATION_NOTES.md`
4. 压成 1 到 2 句写入 `PROGRESS.md`
5. 提炼用户怎么测，写入 `TEST_GUIDANCE.md`
6. 更新 `PROJECT_STATE.md` 的当前阶段、结论和下一步

## 一句话标准

长回执不是要“保存下来”，而是要“按用途拆开，只留下后续判断和测试真正需要的部分”。
