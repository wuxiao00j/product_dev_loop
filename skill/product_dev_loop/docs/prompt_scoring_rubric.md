# Prompt Scoring Rubric

这份文档用于在 checklist 之外，再判断一个 prompt 是不是“写够了”。

原则：

- checklist 解决“有没有”
- scoring 解决“够不够强”

只有两者都过，才建议发出。

## 评分方式

总分 `18` 分，共 `9` 个维度，每个维度按 `0 / 1 / 2` 评分。

- `0 分`：缺失，或写了等于没写
- `1 分`：有，但很弱，执行时仍容易追问
- `2 分`：清楚、可执行、可直接用于派单

## 九个评分维度

### 1. 项目上下文锁定

- `0`：没写项目，或上下文来源不清楚
- `1`：写了项目名，但没有说明当前状态或上轮结论
- `2`：项目名、当前状态、任务来源都清楚

### 2. 轮次 / 阶段 / PRD 状态

- `0`：缺任意关键项
- `1`：写了，但不完整或模糊
- `2`：轮次、阶段、PRD 状态都明确

### 3. 本轮目标是否收敛为单一小闭环

- `0`：目标发散，像任务池
- `1`：有主目标，但仍混入多个动作
- `2`：明显只收敛到一个本轮小闭环

### 4. 必做 / 不做边界

- `0`：没有边界，或只有“尽量别扩散”
- `1`：写了边界，但不够具体
- `2`：必做、不做、保持不动都清楚

### 5. 验收标准是否可判定

- `0`：没有验收标准，或只有空话
- `1`：有验收项，但难以实际判断
- `2`：验收项可直接判断完成与否

### 6. 结构化回传要求是否具体

- `0`：只要求“做完告诉我”
- `1`：有结构，但关键字段不全
- `2`：已完成、未完成、关键代码、特殊状态、风险都明确

### 7. 测试要求是否可执行

- `0`：没写测试要求
- `1`：只写“给测试建议”
- `2`：测试入口、步骤、数据、预期结果、失败判定都明确

### 8. PRD 对照要求是否明确

- `0`：没要求对照 PRD
- `1`：只笼统提到 PRD
- `2`：要求说明已满足项、未满足项和是否达成本轮验收

### 9. 变更建议 / 隐性需求处理规则

- `0`：没写，执行器可能直接改范围
- `1`：有提醒，但规则不完整
- `2`：明确区分变更建议、隐性偏好和 PRD 锁定规则

## 一票否决项

出现任一情况，即使分数勉强够，也不建议发出：

- 没有项目名
- 没有轮次或阶段
- 没有 PRD 状态
- 没有本轮不做
- 没有验收标准
- 没有结构化回传
- 没有测试要求
- 没有 PRD 对照要求

## 等级判断

### 0 到 8 分：不建议发出

表现：

- prompt 仍像普通需求描述
- 执行者大概率需要大量追问
- 很容易扩功能或回传失焦

### 9 到 12 分：仅及格

表现：

- 基本可派，但仍偏松
- 常见问题是边界不够清楚、测试要求太弱、回传不够具体

建议：

- 至少再补强边界、验收或结构化回传中的一项

### 13 到 15 分：可直接发

表现：

- 已能支持真实派单
- 范围、验收、回传口径基本稳定

建议：

- 发出前再用任务类型规则过一遍，确认没有混成“build + fix + polish”杂糅 prompt

### 16 到 18 分：高质量收口型 prompt

表现：

- 执行者看完即可开做
- 主 agent 拿回执后能直接判断是否过关
- 对多轮开发的退化有明显抵抗力

## 使用方式

建议每次生成 prompt 后：

1. 先过 `prompt_quality_checklist.md`
2. 再按本规约打分
3. 低于 `13` 分时，默认不要直接发

## 最低标准

最低建议标准：

- checklist 不出现一票否决项
- 评分至少 `13` 分

如果只到 `9` 到 `12` 分，说明它“能发”，但还不够稳；在真实多轮开发里，仍很容易退化。
