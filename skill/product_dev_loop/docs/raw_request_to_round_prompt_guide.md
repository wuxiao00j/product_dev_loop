# Raw Request To Round Prompt Guide

这份文档专门解决一个常见误判：

- 用户给了一大段需求
- 里面甚至带了文件路径、接口名、页面要求、命令要求
- 但它仍然不是可以直接派给 Cursor、Claude Code 或 Codex 的合格 round prompt

结论先写在前面：

- 用户草稿需求不等于合格派单 prompt
- 需求写得很长，也不等于 prompt 已经收口
- 进入 `Build Delegation` 前，先经过 `Prompt Gate`
- 如果 Gate 结果是 `REWRITE_REQUIRED`，再做 `prompt normalization / prompt rewrite`
- 只有 Gate 最终拿到 `PASS` 后，才允许正式派单

## 1. 什么类型输入属于“原始需求草稿”

原始需求草稿指的是：

- 它主要在描述“想做什么”
- 但没有把“这轮只做什么、这轮不做什么、怎样算完成、做完怎么回传”一起锁住

常见信号：

- 像功能清单，而不是一轮小闭环
- 像 issue 池，而不是当前轮目标
- 像想法块，而不是带边界的执行单
- 像多模块打包任务，而不是单轮可控任务
- 像实现指令，但没有验收和回传要求

至少把下面这些输入都视为“原始需求草稿”：

### 1. 功能清单

常见样子：

- “把登录、个人页、通知、分享都做一下”
- “先把列表、详情、编辑、导出补齐”

问题：

- 它更像 backlog 合集，不像一轮任务

### 2. Issue 列表

常见样子：

- “这几个问题都修一下：空状态、按钮抖动、接口报错、权限判断”

问题：

- 它是问题池，不代表已经判断过本轮先修哪一个

### 3. 用户口述的一段需求块

常见样子：

- 一整段自然语言，夹杂目标、偏好、担心、建议和顺手想法

问题：

- 里面常常同时混着主目标、次要想法和未来需求

### 4. 多模块打包需求

常见样子：

- “把前端页面、接口、埋点、权限、导出、后台配置都一起补掉”

问题：

- 范围过大，无法保证本轮闭环

### 5. 带文件路径但缺边界的任务说明

常见样子：

- “改 `src/pages/a.tsx`、`api/user.ts` 和 `hooks/use-x.ts`，把这个能力做出来”

问题：

- 有落点，不等于有边界
- 给了文件路径，仍可能没有本轮不做、验收和回传

### 6. 带命令 / API / UI 要求但没有验收和回传的开发单

常见样子：

- “接这个 API，命令跑一下，UI 按 Figma 做，做完告诉我改了什么”

问题：

- 只有实现方向，没有完成定义
- “告诉我改了什么” 不等于结构化回传

## 2. 为什么不能直接派给执行器

原始需求草稿不能直接派，不是因为它不够详细，而是因为它缺少执行闭环。

直接派的典型后果：

- 执行器自己挑范围，导致越做越大
- 把多个目标混成一轮，最后什么都做了一点，但没有一个真正闭环
- 在 `PRD.md` 已 `locked` 时偷偷改范围
- 做完只回一句“基本完成”，主 agent 无法判断是否达标
- 没有测试入口、测试数据和失败判定，用户只能盲测
- 多项目并行时串上下文、串文件、串验收口径

所以这里的核心不是“把需求润色一下”，而是把原始草稿收成：

- 可控
- 可验收
- 可测试
- 可回传
- 可对照 PRD

## 3. 原始需求草稿最常见的缺口

常见缺口通常不是一个，而是成组出现：

### 1. 没锁定项目上下文

- 项目名不清楚
- 当前项目目录不清楚
- 多项目并行时不知道这段草稿属于哪个项目

### 2. 没写轮次 / 阶段 / PRD 状态

- 不知道是 `Discovery`、`PRD` 还是 `Build Delegation`
- 不知道 `PRD.md` 目前是 `draft` 还是 `locked`
- 不知道这轮任务来自 PRD、反馈还是上轮未完成项

### 3. 没判断任务类型

- 没区分这轮是 `build`、`fix`、`polish` 还是 `unblock`
- 结果把开发、修复、优化、解阻混写在一起

### 4. 没把目标收成一个可控 round

- 一次塞多个模块
- 一次塞多个页面、接口、命令、体验优化
- 看起来很完整，实际上不可能在一轮里稳定验收

### 5. 没写本轮不做

- 常见后果是“顺手一起补”
- 执行器为了完整性自然扩散

### 6. 没写验收标准

- 只有“做完”或“补齐”
- 没有外部可判断结果

### 7. 没写依赖和降级规则

- 接口是否可用不清楚
- 权限、环境、数据源、第三方服务不确定
- 却没有写“依赖不满足时如何处理 / 什么时候停下”

### 8. 没写结构化回传

- 只有“做完告诉我”
- 没要求写已完成、未完成、关键文件、特殊状态、风险、测试建议、PRD 对照

### 9. 没写测试要求

- 没有测试入口
- 没有操作步骤
- 没有预期结果和失败判定

### 10. 没写 PRD 对照要求

- 做出来的东西可能偏离 PRD
- 主 agent 也无法快速判断本轮到底有没有达标

## 4. skill 应如何把原始草稿改写成合格 round prompt

这份文档默认在 `Prompt Gate = REWRITE_REQUIRED` 之后使用。

改写的目标不是“尽量保留草稿里的所有愿望”，而是把它收成一个本轮可控闭环。

建议固定按下面顺序收口：

### Step 1. 先判断它是不是草稿，而不是合格 prompt

先问：

- 这段输入是在描述想法，还是已经定义好一轮任务
- 有没有项目、轮次、阶段、PRD 状态
- 有没有本轮不做、验收、测试、回传、PRD 对照

只要这些关键项缺一组，就先判为原始需求草稿。

### Step 2. 先锁定项目上下文

至少补齐：

- 项目名
- 当前轮次
- 当前阶段
- 当前 PRD 状态
- 本轮目标来自哪里

如果项目不唯一，先补问，不要默认串项目。

### Step 3. 判断当前轮任务类型

必须先选：

- `build`
- `fix`
- `polish`
- `unblock`

如果一个草稿同时混着多个类型，优先保留本轮主目标，其余内容移到：

- 后续轮次建议
- `CHANGE_PROPOSALS.md`
- `IMPLICIT_REQUIREMENTS.md`

### Step 4. 把草稿压成一个最小闭环

收口规则：

- 只保留本轮最影响闭环的一件事
- 如果草稿过大，先拆分，不强行一轮打包
- 如果一轮做不完，就定义本轮最低闭环
- “看起来很重要但不是本轮验收必需”的内容，先移出本轮

### Step 5. 显式写出本轮不做

至少写清：

- 哪些功能不做
- 哪些模块不动
- 哪些结构、方案或接口不顺手改

这是防扩散的硬边界，不是礼貌提醒。

### Step 6. 补齐验收标准

验收标准必须能回答：

- 用户能看到什么结果
- 哪个主路径必须走通
- 什么情况下仍算未完成

不要写成：

- “体验更好”
- “补齐一下”
- “尽量完善”

### Step 7. 写依赖与降级规则

只要草稿里存在这些不确定项，就必须显式补齐：

- 外部接口
- 权限
- 环境
- 数据源
- 第三方服务
- 等待用户确认的决策

至少说明：

- 依赖满足时怎么做
- 依赖不满足时允许的降级方案
- 哪些情况应直接标记阻塞，不继续扩做

### Step 8. 写结构化回传

至少要求执行器返回：

1. 已完成
2. 未完成
3. 关键文件与职责
4. 关键逻辑
5. `stub / mock / 假数据 / 隐藏入口 / 未接线`
6. 风险与需确认点
7. 测试建议
8. PRD 对照

### Step 9. 写测试要求和 PRD 对照

必须补齐：

- 测试入口
- 操作步骤
- 测试数据
- 预期结果
- 失败判定
- 满足了哪些 PRD 条目
- 还没满足哪些 PRD 条目

### Step 10. 在发出前再过两层检查

先过：

- `docs/prompt_gate_protocol.md`
- `docs/raw_prompt_defect_checklist.md`

再过：

- `templates/prompt_quality_checklist.md`
- `docs/prompt_scoring_rubric.md`

只有 rewrite 完成并重新通过 `Prompt Gate PASS` 后，才进入正式派单。

## 5. 什么情况下即使功能要求很详细，也仍不算合格 prompt

下面这些情况都很常见，而且最容易被误判成“已经写得够细了”：

### 1. 写了很多功能点，但没有单轮目标

- 详细不等于收敛

### 2. 写了很多文件路径，但没有边界

- 落点清楚，不等于范围清楚

### 3. 写了 API、命令和 UI 要求，但没有验收

- 有实现动作，不等于有完成定义

### 4. 写了“做完告诉我”，但没有结构化回传

- 自由汇报不等于可收口

### 5. 写了“自己判断怎么做更好”，但没有 PRD 约束

- 这会把范围判断交给执行器

### 6. 写了“顺便把相关的都补齐”

- 这是最典型的扩散信号

### 7. 草稿本身跨越多个模块、多个子系统、多个阶段

- 即使每一条都合理，也仍不适合直接当单轮 prompt

### 8. `PRD.md` 已 `locked`，但草稿又偷偷改了范围

- 即使写得很完整，也应该先走 `CHANGE_PROPOSALS.md`

## 6. 收口后的最低标准

原始需求草稿只有在被改写成下面这种形态后，才算合格 round prompt：

- 已锁定项目上下文
- 已写明轮次 / 阶段 / PRD 状态
- 已判断任务类型
- 已收成一个本轮小闭环
- 已写本轮不做
- 已写验收标准
- 已写依赖 / 降级规则
- 已写结构化回传
- 已写测试要求
- 已写 PRD 对照要求

如果还只是“需求描述更完整了一点”，但没有补齐以上字段，就仍然不能直接发给执行器。
