# Feedback To Prompt Mapping

这份文档定义：测试反馈、执行器回执和用户反馈，如何转成下一轮 prompt。

核心原则：

- 不是所有反馈都应该继续派单
- 不是所有问题都应该写成 `fix`
- 不是所有用户意见都应该直接改 PRD

先做判断，再写 prompt。

## 统一判断顺序

收到反馈后，先按这个顺序判断：

1. 这是实现缺口，还是范围变化
2. 这是继续派单的问题，还是先等用户确认的问题
3. 这轮应该写成 `build`、`fix`、`polish` 还是 `unblock`
4. 是否需要先进入 `CHANGE_PROPOSALS.md`
5. 是否需要先写入 `IMPLICIT_REQUIREMENTS.md`

## 1. 功能未完成

### 如何识别

- 执行器明确写了 `未完成`
- 验收项只完成一部分
- 主路径还没闭环

### 下一轮 prompt 应强调

- 任务类型通常为 `build`
- 本轮只补剩余最关键闭环
- 明确未完成项来自哪一轮回执
- 明确哪些已完成部分不要重做

### 是否继续派单

- 是

### 是否先等用户确认

- 否

### 是否进入 change proposal

- 否，除非未完成的原因是范围变化

### 是否写入 implicit requirements

- 否

### 是否应禁止继续扩功能

- 是。本轮先补闭环，不扩新功能

## 2. 功能已实现但未接线

### 如何识别

- UI、接口或逻辑单独存在，但没有接到主路径
- 回执出现 `未接线`、隐藏入口、临时入口
- 用户无法从正常路径使用

### 下一轮 prompt 应强调

- 任务类型通常为 `build` 或 `fix`
- 强调接线位置、触发入口和主路径闭环
- 强调不要重做已有部分，只补接线

### 是否继续派单

- 是

### 是否先等用户确认

- 否

### 是否进入 change proposal

- 否，除非接线会改动已锁定范围

### 是否写入 implicit requirements

- 否

### 是否应禁止继续扩功能

- 是。先让已有实现接到主路径

## 3. 只有 UI 壳子，没有真实逻辑

### 如何识别

- 页面能看，但提交、保存、调用、状态流转都没有真实逻辑
- 回执出现 `stub`、`mock`、假数据
- 验收只满足视觉层，不满足行为层

### 下一轮 prompt 应强调

- 任务类型通常为 `build`
- 强调真实逻辑、数据流、输入输出结果
- 验收里要明确“不是只看 UI”

### 是否继续派单

- 是

### 是否先等用户确认

- 否

### 是否进入 change proposal

- 否，除非真实逻辑并不在当前 PRD 范围内

### 是否写入 implicit requirements

- 否

### 是否应禁止继续扩功能

- 是。先把壳子变成可用闭环

## 4. 逻辑已接但数据未落地或未持久化

### 如何识别

- 交互能走通，但刷新后丢失
- 数据只在前端内存、临时状态或 mock 层
- 回执里承认未持久化或临时数据方案

### 下一轮 prompt 应强调

- 任务类型通常为 `build` 或 `unblock`
- 明确本轮只补数据落地或持久化链路
- 区分临时方案和正式方案

### 是否继续派单

- 视 PRD 而定
- 如果当前轮 PRD 就要求真实落地，应继续派单
- 如果 PRD 只要求演示闭环，可先进入测试

### 是否先等用户确认

- 如果补持久化会扩大范围，要先确认

### 是否进入 change proposal

- 当 `PRD.md` 已 `locked` 且持久化不在当前范围内时，进入

### 是否写入 implicit requirements

- 否

### 是否应禁止继续扩功能

- 是。先判断数据落地是不是本轮范围

## 5. 主路径可用，但边界条件失败

### 如何识别

- 正常流程能走通
- 空状态、异常输入、重复提交、错误提示等边界情况失败
- 测试反馈集中在边界条件

### 下一轮 prompt 应强调

- 任务类型通常为 `fix` 或 `polish`
- 写清具体边界条件
- 写清不能破坏主路径
- 验收要包含修后回归检查

### 是否继续派单

- 是

### 是否先等用户确认

- 否，除非反馈实际上引入新需求

### 是否进入 change proposal

- 否，除非边界处理超出 PRD 定义

### 是否写入 implicit requirements

- 如果是稳定偏好类边界处理，先确认后可写

### 是否应禁止继续扩功能

- 是。只修边界，不扩主功能

## 6. 实现与 PRD 不一致

### 如何识别

- 做出来的结果和 PRD 条目不一致
- 执行器回执里承认与 PRD 存在偏差
- 用户反馈“这不是我们之前说的”

### 下一轮 prompt 应强调

- 优先不要急着派单
- 先明确是实现偏差，还是 PRD 需要变更

### 是否继续派单

- 如果是实现偏差，继续派 `fix`
- 如果是需求变更，不应直接继续实现

### 是否先等用户确认

- 是，尤其在 `PRD.md` 已 `locked` 时

### 是否进入 change proposal

- 如果用户想保留当前偏差方向，进入

### 是否写入 implicit requirements

- 只有在确认这是稳定偏好，而不是新范围时才写

### 是否应禁止继续扩功能

- 是。先对齐 PRD，再动代码

## 7. 用户反馈其实是新增需求，不是 bug

### 如何识别

- 当前实现符合 PRD，但用户希望额外增加能力
- 反馈内容改变范围、能力或验收标准

### 下一轮 prompt 应强调

- 不应直接写下一轮开发 prompt
- 先整理成变更提议

### 是否继续派单

- 否，先暂停

### 是否先等用户确认

- 是

### 是否进入 change proposal

- 是

### 是否写入 implicit requirements

- 否，新增需求不等于隐性偏好

### 是否应禁止继续扩功能

- 是

## 8. 用户反馈属于隐性偏好，不应直接改 PRD

### 如何识别

- 用户反复强调“默认应该这样”
- 更像交互习惯、文案偏好、展示偏好，而不是新功能
- 会影响后续判断标准，但不一定立即改变当前范围

### 下一轮 prompt 应强调

- 先不要直接写开发 prompt
- 先确认它是否要成为稳定要求

### 是否继续派单

- 不一定
- 若只是待确认偏好，先暂停
- 若已确认且不改范围，可再写 `polish` 或 `fix`

### 是否先等用户确认

- 是

### 是否进入 change proposal

- 如果它会影响范围或验收标准，进入

### 是否写入 implicit requirements

- 是，确认后写入

### 是否应禁止继续扩功能

- 是

## 9. 当前问题依赖外部条件，应该阻塞等待

### 如何识别

- 缺接口、缺权限、缺环境、缺配置、缺账号、缺用户决策
- 当前代码侧继续做也无法真正闭环

### 下一轮 prompt 应强调

- 若还有可做的解阻动作，写成 `unblock`
- 若没有，就不要继续派开发

### 是否继续派单

- 只有在存在明确解阻动作时才继续

### 是否先等用户确认

- 经常需要

### 是否进入 change proposal

- 通常不需要，除非为绕过阻塞要改范围

### 是否写入 implicit requirements

- 否

### 是否应禁止继续扩功能

- 是。先解除阻塞，别扩范围

## 10. 当前实现已满足本轮目标，可以进入测试或下一阶段

### 如何识别

- 验收项已满足
- 没有阻断闭环的 `stub / mock / 假数据 / 未接线`
- PRD 对照没有关键缺口

### 下一轮 prompt 应强调

- 不一定需要下一轮 prompt
- 应先进入 `Test Guidance` 或 `Feedback Capture`

### 是否继续派单

- 否，除非测试后又发现问题

### 是否先等用户确认

- 否，先测试或收口

### 是否进入 change proposal

- 否

### 是否写入 implicit requirements

- 否，除非测试中确认了新的稳定偏好

### 是否应禁止继续扩功能

- 是。先完成测试与结论判断

## 快速映射结论

- 实现缺口：通常继续派 `build` 或 `fix`
- 体验细节问题：通常派 `polish`
- 环境与依赖卡点：通常派 `unblock`
- 新需求：先停，走 `CHANGE_PROPOSALS.md`
- 隐性偏好：先停，确认后写 `IMPLICIT_REQUIREMENTS.md`
- 已满足本轮目标：不继续扩功能，先测试或收口
