# 示例：从反馈生成下一轮 prompt

## 当前轮反馈

- 项目：`todo-lite`
- 当前轮次：`round-03`
- 当前结论：`继续修改`
- 执行器回执关键信息：
  - 已完成：待办表单和列表展示
  - 未完成：新增后的数据只存在内存中，刷新后丢失
  - 特殊状态：当前保存链路仍是临时假数据
  - PRD 对照：已满足“新增待办”“展示列表”；未满足“刷新后仍可看到新增结果”

## 先判断反馈类型

这类反馈更接近：

- `逻辑已接但数据未落地 / 未持久化`

它不是：

- 纯 UI 问题
- 新增需求
- 隐性偏好

## 下一轮该怎么处理

- 如果当前 PRD 已明确要求真实持久化，继续派单
- 下一轮任务类型优先判断为 `build`
- 本轮禁止继续扩新功能，先把数据落地链路补到能满足当前验收

## 转成下一轮 prompt 的写法示例

你现在要执行 `todo-lite` 项目的 `round-04` 开发任务。本轮类型是 `build`，目标是补齐当前闭环里缺失的数据落地，不扩新功能。

【当前项目信息】
- 项目名：`todo-lite`
- 当前轮次：`round-04`
- 当前阶段：`Build Delegation`
- 当前 PRD 状态：`locked`
- 上轮结论：`Round 03 已完成新增待办与列表展示，但新增结果刷新后丢失，数据仍未落地`

【执行前必读上下文】
- `PROJECT_STATE.md`
- `PRD.md`
- 最近一轮 `PROGRESS.md`
- 最近一轮 `IMPLEMENTATION_NOTES.md`
- `TEST_RESULTS.md`

【本轮目标】
- 本轮只解决：让新增待办的数据落地到当前 PRD 要求的保存链路
- 这轮完成后，用户可以直接看到的结果：新增一条待办后刷新页面，结果仍然存在
- 本轮目标来源：上轮回执中“数据未落地 / 刷新后丢失”问题，以及 PRD 中持久化要求

【本轮边界】
- 必做：补齐保存链路、保住现有新增与展示主路径
- 不做：删除待办、筛选、分享、多人协作
- 保持不动：页面结构和无关模块
- 重点文件 / 模块：`src/services/todo-api.ts`、`src/pages/todos.tsx`

【验收标准】
- 验收项 1：新增待办后刷新页面，结果仍存在
- 验收项 2：本轮不再依赖明显假数据完成保存闭环
- 验收项 3：不引入新的范围变化

【测试要求】
- 请给出测试入口、操作步骤、测试数据、预期结果、失败判定

【完成后必须结构化回传】
1. 已完成
2. 未完成
3. 关键代码信息
4. `stub / mock / 假数据 / 隐藏入口 / 未接线`
5. 风险与需确认点
6. 测试建议
7. PRD 对照
