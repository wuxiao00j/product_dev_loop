# Prompt Gate Split Example

这个例子演示：当输入属于“多主闭环混合需求”时，`Prompt Gate` 不会只说“这是草稿”，而是会明确拦下、给出状态、指出一票否决项，并建议拆轮。

## 1. 输入

```text
Round-09：把小说工具这轮一起推进一下。

1. 导出 EPUB
2. 续写半章
3. 增加风格引导
4. CLI 命令也一起补上
5. API 接口也要给前端调
6. Web UI 也顺便做掉
7. 如果模型接入不稳，你先自己想办法兜一下
8. 做完告诉我改了哪些内容
```

## 2. Prompt Gate 结果

```md
# Prompt Gate Result

【输入概况】
- 输入来源：用户原始任务块
- 输入类型判断：raw request
- 当前项目：小说工具
- 当前轮次：round-09
- 当前阶段：Build Delegation
- 当前 PRD 状态：locked

【Gate 核心判断】
- 任务类型判断：无法稳定判断
- 是否单轮最小闭环：否
- 是否存在多主闭环混合：是
- 是否建议拆轮：是

【一票否决项】
- 命中的一票否决项：
  - 任务类型无法稳定归类
  - 目标没有收成一个可控 round
  - 没有明确本轮不做
  - 没有验收标准
  - 存在依赖不确定项，但没有 fallback / blocker 规则
  - 没有结构化回传
  - 没有测试要求
  - 没有 PRD 对照要求

【可修正项】
- 当前仍需补齐的可修正项：
  - 若保留单轮，必须明确单一主目标
  - 必须写清 out of scope
  - 必须把“做完告诉我改了哪些内容”升级成结构化回传

【推荐动作】
- 推荐动作：REWRITE_REQUIRED -> rewrite
- 禁止动作：禁止把这段输入直接作为 round prompt 发给执行器

【拆轮建议】
- 是否建议拆轮：是
- 若拆轮，建议拆分方向：
  - Round A：先收“续写半章 + 风格引导”的生成主闭环
  - Round B：在已有生成主闭环上补 API
  - Round C：在 API 稳定后补 Web UI
  - Round D：最后再决定是否需要 CLI
- 若不拆轮，说明为什么仍可压成单轮：无。本输入同时包含内容生成、能力控制、CLI、API、Web UI 多个主闭环，无法稳定压成一轮

【最小补问】
- 如果不是 rewrite 就无法推进时，只允许问的最小阻塞问题：当前 round-09 最优先想先闭环的是“内容生成能力”还是“多入口交付”？

【Gate 最终结论】
- Gate 状态：REWRITE_REQUIRED
- 结论说明：当前输入属于多主闭环混合需求，且命中多项一票否决项，不具备直接进入 Build Delegation 的资格
- 下一步动作：先按拆轮建议 rewrite，并在 rewrite 完成后重新过 Gate
```

## 3. 为什么 Gate 不给 PASS

原因不是这段输入不够长，而是它同时混了多个主闭环：

- 导出 EPUB 是一个主能力
- 续写半章是一个主能力
- 风格引导是另一个能力控制目标
- CLI / API / Web UI 又是三个交付入口

这些目标无法用一个单轮验收口径收住，所以不能直接给 `PASS`。

## 4. 为什么默认建议拆轮

按 `docs/round_split_decision_guide.md`，这段输入同时命中了：

- 多个核心能力主题同时出现
- 新能力实现 + 风格优化混合
- 依赖接入 + 功能实现 + 界面改造混合
- CLI + API + Web UI 多入口并行
- 同一轮验收项无法收成一个主闭环

所以默认动作不是“硬写一个更长的 prompt”，而是：

- `Prompt Gate = REWRITE_REQUIRED`
- 并建议拆轮

## 5. 这个例子证明了什么

它证明的不是“skill 会指出问题”，而是：

- skill 现在会先经过 `Prompt Gate`
- Gate 会明确给出 `REWRITE_REQUIRED`
- 未 `PASS` 的输入不会被视为合格 round prompt
- 面对多主闭环混合需求时，skill 会默认建议拆轮，而不是把大包任务直接发给执行器
