你现在要执行一轮已经收口的小步开发任务。下面每一段都不能省；如果某一段写不出来，说明这轮还没准备好，不要直接开做。只有功能描述，不算合格 prompt。

【任务类型｜先判断这轮到底属于哪一类，不要混写】
- 本轮类型：`build / fix / polish / unblock`
- 选择原因：`一句话说明为什么是这个类型`
- 类型提示：
  - `build`：补一个最小闭环，不要铺很多壳子
  - `fix`：修明确问题，要写复现和修后验证
  - `polish`：优化已有路径，要写可动范围和不可动范围
  - `unblock`：先解除阻塞，要写临时方案和正式方案

【当前项目信息｜这段用于锁定项目、轮次、阶段和 PRD 边界，防止串任务】
- 项目名：`<project-name>`
- 当前轮次：`round-0x`
- 当前阶段：`Build Delegation`
- 当前 PRD 状态：`draft / locked`
- 上轮结论或当前状态：`用一句话说明为什么会进入这一轮`

【执行前必读上下文｜这段用于确保你先对齐项目现状，再开始实现】
- 必读：`PROJECT_STATE.md`
- 必读：`PRD.md`
- 必读：最近一轮 `PROGRESS.md`
- 必读：最近一轮 `IMPLEMENTATION_NOTES.md`
- 按需补读：`ARCHITECTURE.md`、`CHANGE_PROPOSALS.md`、`IMPLICIT_REQUIREMENTS.md`、`DECISIONS.md`、`TEST_RESULTS.md`
- 读完后先确认：你理解的当前状态、本轮目标、当前风险
- 如果上下文冲突，先指出冲突，不要直接改代码

【本轮目标｜这段用于把任务收敛成一个本轮能完成、能验证的小目标】
- 本轮只解决：`一句话写清这轮只做什么`
- 这轮完成后，用户可以直接看到的结果：`写结果，不写愿景`
- 本轮目标来源：`PRD 条目 / 测试反馈 / 上轮未完成项`

【类型补强｜按类型把这轮最容易漏掉的约束补上】
- 如果是 `build`：`写本轮最小闭环和关键数据流`
- 如果是 `fix`：`写复现现象、修复目标、回归风险`
- 如果是 `polish`：`写可动范围、不可动范围，防止借机重构`
- 如果是 `unblock`：`写阻塞点、解阻目标、临时方案 / 正式方案`

【本轮边界｜这段用于防止顺手扩功能；如果没有边界，默认就是没收口】
- 必做：`这轮不做完就不算完成的内容`
- 不做：`本轮明确禁止扩散的内容`
- 保持不动：`本轮不要顺手改的页面、接口、目录或方案`
- 重点文件 / 模块：`优先落点`

【验收标准｜这段用于定义“做到什么程度才算这轮完成”】
- 验收项 1：`外部可观察、可判断`
- 验收项 2：`最好直接对应一个用户结果`
- 验收项 3：`如果是修复类任务，要写原问题如何被验证为已消失`

【执行约束｜这段用于防止开发过程悄悄改范围或改定义】
- 只处理本轮目标，不顺手补其他功能
- 不做无关重构，不改无关命名或目录结构
- 如果 `PRD.md` 已 `locked`，不得擅改定义
- 如果发现新需求，只提出变更建议，不直接扩做
- 如果发现隐性要求或隐性偏好，先标记并请求确认，不直接写进 PRD

【测试要求｜这段用于让本轮结果可验证，而不是只靠口头说完成】
- 请返回测试入口
- 请返回操作步骤
- 请返回测试数据
- 请返回预期结果
- 请返回失败判定
- 如果本轮是修复类任务，请说明如何验证原问题已消失且没有明显回归

【完成后必须结构化回传｜这段用于让主 agent 快速收口，而不是看大段过程复述】
请只按下面结构返回，不要粘贴大段原始过程：

1. 已完成
- 实际完成了什么
- 对应满足了哪些验收项

2. 未完成
- 还有哪些没完成
- 为什么没完成

3. 关键代码信息
- 改了哪些关键文件
- 每个关键文件负责什么
- 关键函数、组件、接口或逻辑点是什么

4. 特殊状态检查
- 哪些地方仍然是 `stub`
- 哪些地方仍然是 `mock`
- 哪些地方仍然是 `假数据`
- 哪些入口仍然是 `隐藏入口`
- 哪些逻辑还 `未接线`

5. 风险与需确认点
- 当前已知风险
- 需要用户或产品确认的点
- 如果发现范围变化，请写成变更建议，不要直接改 PRD

6. 测试建议
- 测试入口
- 操作步骤
- 测试数据
- 预期结果
- 失败判定

7. PRD 对照
- 本轮对应哪些 PRD 条目
- 已满足哪些 PRD 条目
- 仍未满足哪些 PRD 条目
- 当前是否达到本轮验收标准
