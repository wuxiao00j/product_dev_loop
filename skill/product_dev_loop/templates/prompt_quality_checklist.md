# Prompt Quality Checklist

在把开发 prompt 发给 Cursor、Claude Code 或 Codex 前，逐项检查。

规则：

- 任何一项明确缺失，都不应视为合格 prompt
- 如果最后只剩功能描述，这份 prompt 直接判定为不合格

## 基础定位

- [ ] 是否先选了任务类型：`build / fix / polish / unblock`
- [ ] 是否写明项目名
- [ ] 是否写明当前轮次
- [ ] 是否写明当前阶段
- [ ] 是否写明当前 PRD 状态

## 上下文要求

- [ ] 是否要求执行前先读 `PROJECT_STATE.md`
- [ ] 是否要求执行前先读 `PRD.md`
- [ ] 是否要求执行前先读最近一轮 `PROGRESS.md`
- [ ] 是否要求执行前先读最近一轮 `IMPLEMENTATION_NOTES.md`
- [ ] 是否说明上下文冲突时先指出，不要直接开做

## 本轮收口

- [ ] 是否写清本轮目标，而不是只有功能描述
- [ ] 是否说明本轮目标来自 PRD、反馈或上轮遗留
- [ ] 是否写了本轮必做
- [ ] 是否写了本轮不做
- [ ] 是否写了重点修改文件、模块或范围
- [ ] 是否按照任务类型补了关键说明

## 验收与完成定义

- [ ] 是否写了可判断的验收标准
- [ ] 是否能从验收标准判断这轮是否完成
- [ ] 是否避免使用“优化一下”“更完整”“体验更好”这类空泛描述

## 结构化回传

- [ ] 是否要求返回 `已完成`
- [ ] 是否要求返回 `未完成`
- [ ] 是否要求返回关键代码信息
- [ ] 是否要求返回关键文件路径与职责
- [ ] 是否要求说明 `stub / mock / 假数据 / 未接线`
- [ ] 是否要求说明隐藏入口或临时方案
- [ ] 是否要求返回风险或待确认点

## 测试要求

- [ ] 是否要求给测试建议
- [ ] 是否要求写测试入口
- [ ] 是否要求写操作步骤
- [ ] 是否要求写测试数据
- [ ] 是否要求写预期结果
- [ ] 是否要求写失败判定

## 类型特有检查

- [ ] 如果是 `build`，是否写了最小闭环或关键数据流
- [ ] 如果是 `fix`，是否写了复现现象、修复目标和修后验证
- [ ] 如果是 `polish`，是否写了可动范围与不可动范围
- [ ] 如果是 `unblock`，是否写了阻塞点，以及临时方案 / 正式方案

## PRD 对照与范围约束

- [ ] 是否要求对照 PRD 判断已满足和未满足项
- [ ] 是否要求说明当前是否达到本轮验收标准
- [ ] 是否明确禁止擅改 PRD
- [ ] 是否说明 PRD `locked` 后范围变化先走 `CHANGE_PROPOSALS.md`

## 新需求与隐性要求

- [ ] 是否说明发现新需求时不能直接扩做
- [ ] 是否说明发现隐性要求时先标记、先确认
- [ ] 是否说明变更建议与实现任务要分开处理

## 一票否决项

只要出现任一情况，就不应发出：

- [ ] 这份 prompt 其实只是“做一个 xx 功能”
- [ ] 没有轮次、阶段或 PRD 状态
- [ ] 没有本轮边界
- [ ] 没有验收标准
- [ ] 没有结构化回传要求
- [ ] 没有测试要求
- [ ] 没有 PRD 对照要求

## 发出前最后一步

- [ ] 是否已经按 `prompt_scoring_rubric.md` 打分
- [ ] 是否达到最低建议标准
