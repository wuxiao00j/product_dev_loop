# 示例：把长回执压缩成项目内关键记录

## 原始执行器回执片段

下面是一段“偏长但常见”的执行器回执风格示例：

> 我已经把 `/todos` 页面的空状态和新增提交流程处理好了。现在进入页面时如果没有待办，会显示空状态提示；提交第一条待办之后，空状态会消失并显示列表。我还顺手整理了一下表单提交逻辑，把一些状态判断合并了。当前列表数据仍然先保存在前端状态里，刷新后不会保留，这部分我先没有接后端，因为看起来这轮应该主要是把空状态补齐。代码主要改了 `src/pages/todos.tsx` 和 `src/components/todo-form.tsx`。理论上现在已经满足本轮目标，不过建议再测一下刷新场景。另外删除功能我没有动，还是未接线状态。

## 第一步：先抽出关键事实

从上面这段里，应优先提炼出这些事实：

- 已完成：空状态展示、提交第一条待办后切换到列表展示
- 未完成：数据刷新后不会保留
- 关键文件：`src/pages/todos.tsx`、`src/components/todo-form.tsx`
- 特殊状态：删除功能仍未接线
- 待验证：执行器认为“理论上满足目标”，但刷新场景仍需重点验证

不应直接保留的原话：

- “我还顺手整理了一下”
- “理论上现在已经满足本轮目标”
- “看起来这轮应该主要是”

## 第二步：写入 `PROGRESS.md`

适合写成：

```md
## 本轮短摘要

- 本轮已补齐 `/todos` 页面的空状态与新增后展示切换，主路径更接近闭环。
- 当前最关键未闭环点是数据刷新后仍会丢失，删除功能也仍未接线。
```

## 第三步：写入 `IMPLEMENTATION_NOTES.md`

适合写成：

```md
## Round 03

- 本轮目标：补齐空状态展示，保持新增主路径可用

### 已完成

- 空列表时展示明确空状态
- 新增第一条待办后，空状态消失并展示列表

### 未完成

- 数据仍只保存在前端状态中，刷新后不会保留

### 关键文件

- 文件：`src/pages/todos.tsx`
- 职责：承载待办页面状态切换与列表展示
- 文件：`src/components/todo-form.tsx`
- 职责：处理待办新增提交

### 关键逻辑

- 页面初始化时根据当前列表状态决定显示空状态还是列表
- 提交第一条待办后，页面从空状态切换到列表展示

### 特殊状态

- `stub`：无明确证据
- `mock`：待确认
- `假数据`：待确认是否仍存在初始化假数据
- `隐藏入口`：无明确证据
- `未接线`：删除功能仍未接线

### 风险

- 执行器声称本轮目标已满足，但刷新后数据丢失仍需重点验证
```

## 第四步：写入 `TEST_GUIDANCE.md`

适合写成：

```md
## 本轮重点验证项

- 空列表时是否能直接看到明确空状态
- 新增第一条待办后是否切换到列表展示
- 刷新页面后新增结果是否仍存在

## 测试入口

- 页面：`/todos`

## 操作步骤

1. 打开 `/todos`
2. 确认初始为空时页面是否显示空状态
3. 输入标题 `买牛奶` 并提交
4. 观察页面是否切换到列表展示
5. 刷新页面
6. 再次观察刚新增的待办是否仍存在

## 预期结果

- 初始为空时显示明确空状态
- 提交后立即看到列表结果
- 如果当前轮真的闭环，刷新后结果仍应保留

## 失败判定

- 页面空状态不清楚
- 提交后没有切换到列表
- 刷新后新增结果丢失

## 已知限制

- 删除功能当前仍未接线，不在本轮验证范围内
- 执行器声称本轮目标已满足，但刷新场景仍需重点验证
```

## 第五步：更新 `PROJECT_STATE.md`

适合写成：

```md
## 最近结论

- 当前结论：`继续修改`
- 最近一轮短摘要：空状态和新增后展示已补齐，但刷新后数据仍会丢失，本轮尚未完全闭环。

## 当前阻塞

- 数据未落地，刷新后结果丢失

## 下一步

- 先根据 `TEST_GUIDANCE.md` 重点验证刷新场景
- 如果确认刷新后仍丢失，则下一轮继续聚焦数据落地，不扩新功能
```

## 这个示例要表达的重点

- 长回执不是直接存档
- `PROGRESS.md` 只写最关键进展
- `IMPLEMENTATION_NOTES.md` 记录实现事实
- `TEST_GUIDANCE.md` 转成用户可操作步骤
- `PROJECT_STATE.md` 只保留当前阶段结论和下一步
