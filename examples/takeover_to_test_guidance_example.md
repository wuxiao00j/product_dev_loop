# 示例：接管后进入“生成用户测试说明”

## 用户最初怎么说

`继续项目 demo，给我一版现在能直接发给老板的测试说明。`

## agent 如何触发最小接管

先读：

1. `docs/agent_handoff_quickstart.md`
2. `projects/demo/PROJECT_STATE.md`
3. `projects/demo/PRD.md`
4. `projects/demo/IMPLEMENTATION_NOTES.md`
5. `projects/demo/TEST_RESULTS.md`

## 接管摘要里识别出的关键结论

- 当前阶段更接近 `Test Guidance`
- 执行器声称主流程已完成
- 当前有明确测试入口
- 但“刷新后是否保留数据”仍缺正式验证

## 为什么下一步该进入这个分流

- 当前最缺的是用户可直接照着测的步骤
- 这时不该先继续派单，因为是否需要继续改还取决于测试结果
- 这时也不该先更新长期记录，因为关键风险已经清楚，直接转成测试说明更有价值

## 第一条正式动作输出应该长什么样

`我先把这轮测试说明整理成用户可读版本，重点会写测试入口、步骤、预期结果，以及“刷新后数据保留”这个执行器声称已完成但仍需重点验的点。`
