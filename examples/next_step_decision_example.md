# 示例：当前不该继续派单，而该等待确认或阻塞等待

## 场景 A：等待用户确认

### 当前情况

- 当前实现已符合 `PRD.md`
- 用户反馈：希望增加“删除待办”
- `PRD.md` 当前状态：`locked`

### 不该直接做什么

- 不应直接继续派单开发删除功能

### 适合输出的下一步建议

```md
## 当前判断

- 建议动作：`先确认`
- 原因：用户提出的是新增需求，不是当前实现缺陷；`PRD.md` 已 `locked`，不能直接扩做

## 当前最关键未闭环点

- 当前轮目标本身已接近完成，但“是否加入删除功能”属于范围变化，尚未确认

## 当前不建议做什么

- 不建议直接进入下一轮开发删除功能

## 下一步

- 先把“删除待办”写入 `CHANGE_PROPOSALS.md`
- 等用户确认是否纳入当前版本
```

## 场景 B：阻塞等待外部条件

### 当前情况

- 当前页面保存逻辑需要真实接口
- 上游接口地址和权限都还没给
- 代码侧没有明确可执行的解阻动作

### 不该直接做什么

- 不应为了“先做点什么”而重做整套数据层

### 适合输出的下一步建议

```md
## 当前判断

- 建议动作：`阻塞等待`
- 原因：当前缺少真实接口地址和调用权限，继续改代码也无法完成真实验证

## 当前最关键未闭环点

- 保存链路依赖的外部接口不可用

## 当前不建议做什么

- 不建议继续派单做大规模改造
- 不建议用会改变已锁定范围的临时绕路方案

## 下一步

- 等待外部接口地址
- 等待调用权限或测试账号
- 条件齐备后再判断是继续 `build` 还是写成 `unblock`
```

## 这个示例强调什么

- 不是“还能不能改”，而是“现在最正确的下一步是什么”
- 主控 agent 要能主动阻止无效继续派单
