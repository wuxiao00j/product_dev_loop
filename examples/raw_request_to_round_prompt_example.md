# Raw Request To Round Prompt Example

这个例子演示：用户给的是一段“看起来很详细”的开发草稿，但 skill 不会直接把它发给执行器，而是会先经过 `Prompt Gate`，拿到 `REWRITE_REQUIRED` 后再收口成合格 round prompt。

## 1. 原始需求草稿

```text
项目还是 ops-console，应该已经到开发阶段了。你把工单列表这块一起补一下：

1. 列表页要能查数据、搜关键字、按状态筛选
2. 点进去要能看详情
3. 最好顺手把导出 CSV 也做了
4. 空状态、loading、报错态都优化一下
5. 如果后端接口不全，你就自己补一下 /api/tickets、/api/tickets/:id
6. 相关文件大概在 src/pages/tickets/*、src/components/ticket-*、server/routes/tickets.ts
7. 如果发现权限判断没接，也顺便补齐
8. 做完告诉我改了哪些文件、怎么测试

尽量一次做完整一点，不要只做一半。
```

## 2. Prompt Gate 先给出的判断

这段草稿不能直接发，原因不是它太短，而是它先被 `Prompt Gate` 判定为 `REWRITE_REQUIRED`。

Gate 看到的关键问题：

- 只有项目名线索，没有明确轮次、阶段和 PRD 状态
- 同时混着 `build`、`polish` 和潜在 `unblock`
- 目标没有收成一个本轮小闭环，而是把列表、搜索、筛选、详情、导出、权限、接口补齐打包在一起
- 给了文件路径，但没有本轮不做，所以落点清楚不等于边界清楚
- 写了“做完告诉我改了哪些文件、怎么测试”，但没有结构化回传和完整测试要求
- 提到了“如果后端接口不全你就自己补”，但没有写依赖不满足时的降级规则和阻塞规则
- 没有 PRD 对照要求

按 `docs/raw_prompt_defect_checklist.md` 判断，这至少触发了这些一票否决项：

- 没有轮次 / 阶段 / PRD 状态
- 没把目标收成一个可控 round
- 没写本轮不做
- 存在依赖不确定项，但没有降级规则
- 没有结构化回传
- 没有 PRD 对照要求

所以 Gate 不会给 `PASS`，后续动作只能是：

- 先 rewrite
- rewrite 后重新过 Gate
- 未重新拿到 `PASS` 前，不进入 `Build Delegation`

## 3. Gate 之后 skill 如何收口

收口思路不是“把原文润色一下”，而是先做范围裁剪：

- 保留当前最影响闭环的一件事：工单列表真实数据接线和状态补齐
- 延后详情、导出、权限、搜索、筛选等内容，不混进本轮
- 明确如果现有数据依赖不可用，就标记阻塞，不允许顺手设计一整套新接口
- 把“做完告诉我”改成结构化回传
- 补齐测试要求和 PRD 对照要求

这样处理后，这轮才会变成一个可控 round，而不是“尽量一次做完整”的大包任务。

## 4. rewrite 并重新过 Gate 后的 round prompt

```text
你现在要执行 ops-console 项目的 round-06 开发任务。本输入已通过 Prompt Gate = PASS。本轮类型是 build，目标已经收口。用户给过一段较大的功能草稿，但这轮只处理“工单列表真实数据接线 + 空状态 / 错误状态闭环”，不要把详情、导出和权限补齐一起混进来。

【任务类型】
- 本轮类型：build
- 选择原因：当前最阻碍闭环的不是新增更多功能，而是让工单列表从真实数据到结果展示先走通

【当前项目信息】
- 项目名：ops-console
- 当前轮次：round-06
- 当前阶段：Build Delegation
- 当前 PRD 状态：locked
- 上轮结论：Round 05 已完成工单列表基础 UI，但当前仍依赖示例数据，空状态和报错态未按 PRD 落地

【执行前必读上下文】
- 必读：PROJECT_STATE.md
- 必读：PRD.md
- 必读：最近一轮 PROGRESS.md
- 必读：最近一轮 IMPLEMENTATION_NOTES.md
- 按需补读：ARCHITECTURE.md、CHANGE_PROPOSALS.md、DECISIONS.md、TEST_RESULTS.md
- 如果上下文冲突，先指出冲突，不要直接开做

【本轮目标】
- 本轮只解决：把工单列表页接到现有真实列表数据来源，并补齐列表页的空状态、loading 和错误状态
- 这轮完成后，用户能直接看到的结果：进入工单列表页后，不再依赖示例假数据；有数据时展示真实列表，无数据时展示明确空状态，请求失败时展示可理解错误状态
- 本轮目标来源：PRD.md 中“工单列表主路径可用”条目，以及 Round 05 回执中的“仍有 mock 数据痕迹、状态页未补齐”遗留

【本轮边界】
- 必做：去掉工单列表的示例假数据依赖；接好现有工单列表数据请求链路；补齐列表页空状态、loading、错误状态
- 不做：工单详情页；导出 CSV；权限体系补齐；搜索和筛选增强；新增后端接口设计
- 保持不动：现有详情页路由结构；非工单模块页面；无关目录和命名整理

【验收标准】
- 验收项 1：进入工单列表页时，不再展示示例假数据，而是展示真实列表结果或明确空状态
- 验收项 2：列表请求进行中时展示 loading 状态；请求失败时展示可理解错误状态，不是空白页
- 验收项 3：本轮不扩做详情、导出、权限或额外接口改造

【依赖与降级规则】
- 关键依赖：现有工单列表接口或现有数据访问层必须能返回真实列表结果
- 依赖满足时：按现有数据链路完成列表页接线和状态展示
- 依赖不满足时允许的降级方案：仅允许把失败状态和缺失依赖明确暴露出来，并在回传里标记阻塞；不要自行发明新的正式接口或扩做详情能力
- 哪些情况应直接标记阻塞，不继续扩做：如果现有列表数据源不可用、接口契约缺失或权限前置条件缺失，直接标记阻塞并说明缺什么

【PASS Evidence】
- Prompt Gate：PASS
- Round main closure：工单列表从真实数据请求到列表 / 空状态 / 错误状态展示形成一个主闭环
- Why not split：空状态、loading、错误状态只是同一列表主路径的必要接线，不是独立闭环
- Fallback / blocker rule：如果现有列表数据源不可用或接口契约缺失，直接标记阻塞，不扩做详情、导出或权限
- Delegation contract：satisfied

【测试要求】
- 请返回测试入口、操作步骤、测试数据、预期结果、失败判定

【完成后必须结构化回传】
1. 已完成
2. 未完成
3. 关键代码信息
4. stub / mock / 假数据 / 隐藏入口 / 未接线
5. 风险与需确认点
6. 测试建议
7. PRD 对照
```

## 5. 为什么这样收口更稳

这次收口真正解决的不是“写得更正式”，而是三件事：

- 把多模块打包任务压成一个单轮最低闭环，避免范围扩散
- 把依赖不确定项补成显式降级规则，避免执行器自己改方案
- 把验收、测试、PRD 对照和 PASS evidence 写进 prompt，避免“看起来做了很多，但主 agent 仍无法判断是否达标”

所以这个例子证明的不是“文档更多了”，而是：

- skill 会先给出 Gate 结论，而不是直接写 prompt
- skill 更会识别不合格 prompt 草稿
- skill 更会把用户需求清单收口成合格 round prompt
- skill 更能防止“看起来很详细，但实际上不可控”的派单直接发给执行器
